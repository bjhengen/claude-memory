# Claude Memory v2 Design

**Date:** 2026-02-15
**Status:** Approved
**Approach:** Hybrid (repo sync + light restructure + incremental features)

## Goals

1. **Richer session context** - Project-level CLAUDE.md for philosophy, scope, and conventions
2. **Lesson lifecycle** - Update and retire lessons to keep search results clean
3. **Project name normalization** - Alias table + case-insensitive lookup to prevent duplicates
4. **Repo hygiene** - Sync deployed OAuth code back to git, establish migrations pattern
5. **Maintainable structure** - Split monolithic server.py into modules

## Non-Goals

- No embedding model changes (still ada-002)
- No auto-dedup or staleness detection
- No auth changes (OAuth stays as deployed)
- No index type changes (ivfflat stays until data volume justifies HNSW)

## Phases

### Phase 0: Database Backup

1. `pg_dump` the live database on EC2 to a timestamped SQL file
2. Copy the dump to local machine as a second copy
3. Verify the dump is restorable

### Phase 1: Repo Sync

1. Pull current deployed `server.py` (with OAuth) from EC2
2. Pull any other drifted files (requirements.txt, docker-compose.yml)
3. Diff against repo, commit deployed state as "sync production to repo"
4. This becomes the baseline for all v2 work

### Phase 2: Restructure

Split `src/server.py` (~1210 lines) into modules:

```
src/
├── __init__.py
├── server.py          # App setup, middleware, lifespan, health checks (~100 lines)
├── config.py          # Environment vars, security settings (~20 lines)
├── db.py              # Embedding helpers, DB pool management (~30 lines)
├── tools/
│   ├── __init__.py
│   ├── search.py      # search, search_lessons
│   ├── projects.py    # get_project, list_projects, CLAUDE.md tools
│   ├── infra.py       # get_connectivity, list_machines, add_machine, add_container
│   ├── lessons.py     # log_lesson, log_pattern, update_lesson, retire_lesson
│   ├── sessions.py    # start_session, end_session
│   ├── journal.py     # write_journal, read_journal
│   └── admin.py       # update_project_state, check_guardrails, get_permissions, merge_projects
└── auth.py            # APIKeyAuthMiddleware + OAuth provider
```

Key constraints:
- Pure refactor, zero behavior change
- Each tool module imports the shared `mcp` instance and registers tools via `@mcp.tool()`
- Same entrypoint: `uvicorn src.server:app`
- Same Docker container, same ports

Add `migrations/` directory:
```
migrations/
├── 001_initial_schema.sql    # Copy of current db/schema.sql (reference)
├── 002_v2_features.sql       # v2 schema changes
```

### Phase 3: Schema Migration

```sql
-- migrations/002_v2_features.sql

-- Project CLAUDE.md
ALTER TABLE projects ADD COLUMN claude_md TEXT;

-- Lesson lifecycle
ALTER TABLE lessons ADD COLUMN retired_at TIMESTAMP;
ALTER TABLE lessons ADD COLUMN retired_reason TEXT;

-- Project name normalization
CREATE TABLE project_aliases (
    id SERIAL PRIMARY KEY,
    alias VARCHAR(100) UNIQUE NOT NULL,
    project_id INT REFERENCES projects(id) ON DELETE CASCADE,
    created_at TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_project_aliases_alias ON project_aliases(LOWER(alias));
```

### Phase 4: New Tools

#### Project CLAUDE.md Tools

`get_project_claude_md(project: str) -> str`
- Returns the CLAUDE.md content for a project
- Called at session start alongside `get_project`
- Returns empty/null gracefully if none set

`set_project_claude_md(project: str, content: str) -> str`
- Creates or fully replaces CLAUDE.md content
- For initial authoring or full rewrites

`update_project_claude_md(project: str, section: str, content: str) -> str`
- Updates a specific section by heading match (e.g., `## Architecture`)
- If section exists, replaces it; if not, appends it
- For incremental updates without rewriting the whole doc

#### Lesson Lifecycle Tools

`update_lesson(lesson_id: int, title: str = None, content: str = None, tags: list[str] = None, severity: str = None) -> str`
- Partial update of any combination of fields
- Regenerates embedding if title or content changes
- Preserves fields not passed

`retire_lesson(lesson_id: int, reason: str = None) -> str`
- Soft delete via `retired_at` timestamp + optional reason
- Retired lessons excluded from search by default
- Data preserved, just hidden

Search updates:
- `search()` and `search_lessons()` add `WHERE retired_at IS NULL`
- `search_lessons` gets optional `include_retired: bool = False` parameter

Retirement is manual/opportunistic only - Claude or Brian retires lessons when encountering stale ones during normal work.

#### Project Name Normalization

All project lookups use a shared helper function:
1. Case-insensitive match on `project_aliases.alias`
2. Fall back to case-insensitive match on `projects.name`
3. Used consistently across all tools that accept a project name

`merge_projects(keep: str, merge: str) -> str`
- Reassigns all lessons, sessions, key_files, journal entries, approaches, project_state from merge to keep
- Creates alias so old name still resolves
- Deletes the merge project record
- Returns summary of what was moved

### Phase 5: Data Cleanup

- Merge duplicate projects (e.g., "Dungeon Days" into "dungeondays")
- Seed aliases for common variations of existing projects

### Phase 6: Deploy & Verify

1. Rebuild container with new code structure
2. Apply migration `002_v2_features.sql`
3. Restart containers
4. Verify all existing tools still work
5. Test new tools
6. Estimated downtime: ~2-3 minutes

## External Interface

No changes. All existing clients (Claude Code, Claude Desktop, Mobile) continue to work without reconfiguration:
- Same URL: `memory.friendly-robots.com/mcp`
- Same auth: Bearer token + OAuth
- Same tool names and parameters
- New tools are additive only

## Relationship: CLAUDE.md vs Structured Data

Project CLAUDE.md holds narrative context:
- Project scope and purpose
- Architecture overview and key conventions
- What Claude should/shouldn't do
- Active development context

Structured tables hold facts:
- `tech_stack`, `key_files`, `approaches`, `guardrails`
- Connectivity (machines, containers, databases)
- Session history, lessons, patterns

A session typically calls both `get_project` (facts) and `get_project_claude_md` (context).
